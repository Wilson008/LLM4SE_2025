## Grammar 2 vs Grammar 1 — differences

### 1) `MongoBean` header changed (keyword removed)
- **Grammar 1:**
  ```xtext
  MongoBean:
    'mongobean' name=ValidID '{' ...
  ```
- **Grammar 2:**
  ```xtext
  MongoBean:
    name=ValidID '{' ...
  ```
**Effect:** In Grammar 2, a bean declaration no longer starts with the literal keyword `mongobean`. Any construct that matches `name=ValidID '{' ... '}'` is a `MongoBean`.
This is a significant syntactic change and can introduce ambiguity with other brace-based constructs if they can start similarly.

### 2) `MongoProperty` type can now be inline-defined (`inlineType=MongoBean`)
- **Grammar 1:**
  ```xtext
  MongoProperty:
    type=JvmTypeReference (many?='*')? name=ValidID;
  ```
- **Grammar 2:**
  ```xtext
  MongoProperty:
    (type=JvmTypeReference | inlineType=MongoBean) (many?='*')? name=ValidID;
  ```
**Effect:** A property’s “type” can now be either:
- a normal JVM type reference (`String`, `java.util.List<Foo>`, etc.), **or**
- an **inline MongoBean** definition used as the property type.

This enables nested/anonymous bean-like type definitions directly where a type is expected.

### 3) What stayed the same
- File structure (`MongoFile`, `Import`, `PackageDeclaration`, `AbstractElement`) is unchanged.
- Operations (`MongoOperation`) are unchanged.
- `QualifiedNameWithWildcard` unchanged.
- Multiplicity marker `*` on properties remains the same.

## Impact on Instance 1 (Grammar 1 instance)
Instance 1 uses `mongobean Artist { ... }` etc. That **matches Grammar 1**, but under **Grammar 2 it would no longer parse as written**, because the `mongobean` keyword is no longer part of `MongoBean`. You’d have to remove `mongobean` (or reintroduce it) for those declarations to match Grammar 2.
