### Differences: Grammar 2 vs Grammar 1

1. **`mongobean` keyword removed from `MongoBean`**
   - **Grammar 1:**
     ```xtext
     MongoBean:
       'mongobean' name=ValidID '{' ... '}';
     ```
   - **Grammar 2:**
     ```xtext
     MongoBean:
       name=ValidID '{' ... '}';
     ```
   **Impact:** In Grammar 2, bean declarations no longer start with the literal keyword `mongobean`. So `mongobean Artist { ... }` (Instance 1 style) would not match Grammar 2; it would instead expect `Artist { ... }`.

2. **`MongoProperty` can use an inline `MongoBean` as a type**
   - **Grammar 1:** properties have only a JVM type:
     ```xtext
     MongoProperty:
       type=JvmTypeReference (many?='*')? name=ValidID;
     ```
   - **Grammar 2:** properties can have either a JVM type reference **or** an inline bean definition:
     ```xtext
     MongoProperty:
       (type=JvmTypeReference | inlineType=MongoBean) (many?='*')? name=ValidID;
     ```
   **Impact:** Grammar 2 allows nested/inline bean structures in property type position, e.g. something conceptually like:
   ```
   Address { String street } address
   ```
   (since `inlineType=MongoBean` is now permitted where a type would be).

3. **Everything else is effectively unchanged**
   - `MongoFile`, `Import`, `AbstractElement`, `PackageDeclaration`, `AbstractFeature`, `MongoOperation`, and `QualifiedNameWithWildcard` are structurally the same (except that `MongoBean` itself changed, which indirectly affects where it is referenced—e.g., inline usage in properties).  

### Note about Instance 1 vs Grammar 2
- Instance 1 includes `mongobean Artist { ... }`, which matches **Grammar 1** but would **not** parse under **Grammar 2** because Grammar 2’s `MongoBean` rule no longer consumes the `'mongobean'` token.
