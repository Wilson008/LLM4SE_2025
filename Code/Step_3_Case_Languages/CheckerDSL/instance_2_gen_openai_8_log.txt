## Differences: Grammar 2 vs Grammar 1

### 1) `ChkVariableDeclaration` syntax changed
**Grammar 1**
```antlr
type=ChkTypeReference (arr+='[' (sz+=CHK_NUMBER)? ']')* name=ValidID
...
('value' '{' spValue=SpValue '}')?;
```
- Allows multiple bracket pairs, and each `[]` may omit a size (`[]` or `[sz]`).
- Uses keyword **`value`** for the special/seed values block.

**Grammar 2**
```antlr
type=ChkTypeReference ('[' sz+=CHK_NUMBER ']')* name=ValidID
...
('sp' '{' spValue=SpValue '}')?;
```
- Each bracket pair **must** include a size: only `[sz]`, not empty `[]`.
- Keyword changed from **`value`** → **`sp`**.

Impact on Instance 1:
- Declarations like `int[n] c` still fit (size is present).
- Any declaration using unsized `[]` would no longer parse in Grammar 2.
- Any use of `value { ... }` must become `sp { ... }`.

---

### 2) `Helper.select` updated (added range after `:`)
**Grammar 1**
```antlr
select='select' '(' vars+=ValidID (',' vars+=ValidID)* ')'
```

**Grammar 2**
```antlr
select='select' '(' vars+=ValidID (',' vars+=ValidID)* ':' ((var3=ChkVariables) '..' (var4=ChkVariables)) ')'
```

- Grammar 2 requires `select(...)` to include a **range** using `: a..b`.
- Instance 1 examples like:
  - `select(k,l) where ...`
  - `select(k) where ...`
  would **not parse** under Grammar 2 unless rewritten like:
  - `select(k,l : 1..n) where ...` (example shape)

---

### 3) `FormatExpression` redesigned + new `FormatVariable`
**Grammar 1**
```antlr
FormatExpression:
  num=CHK_NUMBER '(' (var+=ValidID ('[' sz+=CHK_NUMBER ']')* ('<' (count+=CHK_NUMBER) '>')?
    (',' var+=ValidID ('[' sz+=CHK_NUMBER ']')* ('<' (count+=CHK_NUMBER) '>')?)*)? ')';
```
- Format variables are `ValidID` plus optional indexing `[...]` (with sizes) and optional `<count>`.

**Grammar 2**
```antlr
FormatExpression:
  num=CHK_NUMBER '(' (var+=FormatVariable (',' var+=FormatVariable)*)? ')';

FormatVariable:
  fv=ChkVariable ('<' (count=CHK_NUMBER) '>')?;
```
Key changes:
- A format “variable” is now a full **`ChkVariable`** (optionally prefixed `in./out./ans.`, may include indices, may include backtick newtest marker), not just `ValidID`.
- Indexing is no longer described inside `FormatExpression` via `ValidID ('[' ... ']')*`; it’s inherited from `ChkVariable`.
- `<count>` remains supported.

Impact on Instance 1:
- `n (c[i],p[i],t[i])` is likely still fine because `c[i]` etc. are `ChkVariable`-shaped.
- `1 (y<n>)` becomes: variable `y` with `<n>` count; still matches via `FormatVariable`.

---

### 4) No other structural changes
Everything else (MR structure, FollowUp/Property blocks, expressions, loops, operators, helper set aside from `select`, etc.) is effectively unchanged between the two versions.
