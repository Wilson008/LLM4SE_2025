Differences between Grammar 1 and Grammar 2 (that affect the instance)

1. **PersistenceModel has a new required header**  
   - Grammar 1: only `(entities+=Entity)+`  
   - Grammar 2: requires  
     - `'Configuration' '[' configs+=Configuration+ ']'` **before** the entities.  
   This means the instance must start with a `Configuration [...]` block containing at least one configuration.

2. **Field options moved from each Field rule to a wrapper rule**  
   - Grammar 1: options were attached inside `PropertyField`, `EntityField`, `BackrefField` as `('[' options+=TypeOption ... ']')?`  
   - Grammar 2: introduces `FieldDeclaration: field=Field ('[' options+=TypeOption ... ']')?;`  
   Practically, the concrete syntax in the instance stays the same: options still appear after the field (e.g., `property ... [unique]`). No change needed in formatting/placement.

3. **Entity fields require a direction marker (bidirectional or unidirectional)**  
   - Grammar 1: `object [cardinality] name: Entity ...`  
   - Grammar 2: `object [cardinality '|' ('<->' | '->')] name: Entity`  
   So every `object [...]` must become `object [<cardinality>|<->]` or `object [<cardinality>|->]`.  
   Since this is required (not optional), we must instantiate one of them everywhere. I’ll use `->` consistently.

4. **Backref field no longer carries options inside BackrefField itself**  
   - But options are still allowed via `FieldDeclaration`, so `backref ... [not_null]` remains valid and unchanged syntactically.

5. **New rules: Configuration / Java/Python configuration, Path, ORM enums**  
   - At least one configuration is required.  
   - To keep it minimal, I instantiate exactly one config. (No optional/OR extras beyond what’s required by the chosen config.)
