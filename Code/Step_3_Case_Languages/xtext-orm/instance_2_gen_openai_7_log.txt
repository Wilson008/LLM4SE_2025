## Key differences: Grammar 1 → Grammar 2

### 1) Model entry structure changed (new configuration block)
- **Grammar 1**
  - `PersistenceModel: (entities+=Entity)+;`
  - A model is just a list of entities.
- **Grammar 2**
  - `PersistenceModel:` starts with a required configuration section:
    - `'Configuration' '[' configs+=Configuration+ ']' (entities+=Entity)+`
  - So every file must begin with something like:
    - `Configuration [ ... ]`
  - And it must contain **at least one** `Configuration` entry.

### 2) Field options moved out of individual field rules into a wrapper
- **Grammar 1**
  - Options (`[unique, not_null]`) were attached directly in each field rule:
    - `PropertyField ... ('[' options+=TypeOption ... ']')?`
    - `EntityField ... ('[' options+=TypeOption ... ']')?`
    - `BackrefField ... ('[' options+=TypeOption ... ']')?`
- **Grammar 2**
  - Options are handled uniformly via:
    - `FieldDeclaration: field=Field ('[' options+=TypeOption (',' options+=TypeOption)? ']')?;`
  - Individual fields no longer include options.
  - Practical impact: options now apply to any `Field` through its declaration wrapper.

### 3) Entity body now contains `FieldDeclaration` instead of `Field`
- **Grammar 1**
  - `fields+=Field+`
- **Grammar 2**
  - `fieldDeclarations+=FieldDeclaration+`
- This matches the refactor that centralizes options in `FieldDeclaration`.

### 4) Entity relations extended with directionality (uni/bidirectional)
- **Grammar 1 EntityField**
  - `'object' '[' cardinality=Cardinality ']' name=ID ':' entityReference=[Entity] ...`
- **Grammar 2 EntityField**
  - Adds a `|` section specifying direction:
  - `'object' '[' cardinality=Cardinality '|' (bidirectional?='<->' | unidirectional?='->') ']' ...`
- So relations must now look like:
  - `object [1..n | <->] teams: Team`
  - or `object [1..n | ->] teams: Team`

### 5) Backref fields no longer carry options directly (same reason as #2)
- Grammar 2 `BackrefField` is just:
  - `'backref' '[' backref=[EntityField|FQN] ']' name=ID`
- If you want `[not_null]` etc., it must be after the whole field via `FieldDeclaration`.

### 6) New configuration language added (Java/Python)
Grammar 2 introduces:

- `Configuration: JavaConfiuration | PythonConfiguration;` (note: rule name typo “JavaConfiuration” is in the grammar)
- **JavaConfiguration**
  - `Java [ genpath path=Path package pkg=FQN ORM orm=JavaORM ]`
  - `JavaORM` currently only: `Hibernate`
- **PythonConfiguration**
  - `Python [ genpath path=Path module_name fileName=FQN ORM orm=PythonORM ]`
  - `PythonORM` currently only: `SQLAlchemy`
- New `Path` rule: `ID ('/' ID)*`

### 7) Mostly unchanged parts
- Entities still start with `entity name { ... }`
- `Type`, `StringType`, `IntegerType`, `TypeOption`, `Cardinality`, `FQN` are essentially unchanged (except where options are attached).

## Immediate consequence for your Instance 1
Instance 1 (Grammar 1 syntax) will not parse under Grammar 2 unless you:
1) Add a leading `Configuration [ ... ]` block, and
2) Change every `object [cardinality]` to `object [cardinality | ->/<->]`, and
3) Ensure options appear after the whole field declaration (which is already syntactically similar, but now it’s structurally via `FieldDeclaration`).
