### Differences: Grammar 2 vs Grammar 1

#### 1) Top-level elements allowed inside `Contract`
- **Grammar 1:**
  `(attributes+=Attribute | events+=Event | methods+=Method | modifiers+=Modifier)*`
- **Grammar 2:**
  `(attributes+=Attribute | events+=Event | methods+=Method | validators+=Validator | dataStores+=DataStore)*`

So:
- `Modifier` definitions at contract level are **replaced** by `Validator` definitions.
- A new top-level construct **`DataStore`** is added.

#### 2) `Modifier` renamed/replaced by `Validator`
- **Grammar 1 rule:** `Modifier: 'Modifier' name=ID ...`
  - Keyword is **capitalized**: `'Modifier'`
- **Grammar 2 rule:** `Validator: 'validator' name=ID ...`
  - Keyword is **lowercase**: `'validator'`

Structure inside the block remains the same:
- `message: STRING`
- `validation: STRING`

#### 3) Method references: `modifiers:` becomes `validators:`
Inside `Method`:
- **Grammar 1:**
  `(modifiersKeyword='modifiers:' modifiers+=[Modifier] (',' modifiers+=[Modifier])*)?`
- **Grammar 2:**
  `(modifiersKeyword='validators:' validators+=[Validator] (',' validators+=[Validator])*)?`

Notes:
- The assignment name `modifiersKeyword` is still used, but it now captures the literal `'validators:'` (a naming inconsistency, but not necessarily a parsing problem).
- Cross-references change from `[Modifier]` to `[Validator]`.

#### 4) New `DataStore` rule
Added in Grammar 2:
```xtext
DataStore:
  'dataStore' name=ID 'fromType' fromType=DataType 'toType' toType=DataType
;
```
This enables declarations like:
`dataStore balances fromType address toType integer`

#### 5) Everything else is effectively unchanged
- `Contract` header (`Contract name { ... }`), `version`, optional `hasReceive` and `ownership`
- `Attribute`, `Event`, `Param`, `Output`
- `DataType` enum stays the same (including `array`)
