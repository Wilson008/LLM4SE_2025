## Key differences: Grammar 1 → Grammar 2

### 1) Top-level elements inside `Contract`
- **Grammar 1** allows zero or more of:
  - `attributes+=Attribute | events+=Event | methods+=Method | modifiers+=Modifier`
- **Grammar 2** allows zero or more of:
  - `attributes+=Attribute | events+=Event | methods+=Method | validators+=Validator | dataStores+=DataStore`

So:
- `Modifier` entries were **removed** as a top-level construct and replaced by `Validator`.
- A new top-level construct `DataStore` was **added**.

### 2) `Modifier` renamed/replaced by `Validator`
- **Grammar 1** rule name/keyword: `Modifier: 'Modifier' name=ID ...`
- **Grammar 2** rule name/keyword: `Validator: 'validator' name=ID ...`

Differences:
- Keyword changed from **`Modifier` (capital M)** to **`validator` (lowercase)**.
- The assignment in `Contract` changes from `modifiers+=Modifier` to `validators+=Validator`.

The internal structure is otherwise essentially the same:
- Both have params, and both contain:
  - `message: STRING`
  - `validation: STRING`

### 3) Method’s reference section changed: `modifiers:` → `validators:`
In `Method` body:
- **Grammar 1**
  - `modifiersKeyword='modifiers:' modifiers+=[Modifier] ...`
- **Grammar 2**
  - `modifiersKeyword='validators:' validators+=[Validator] ...`

So in instances:
- `modifiers: onlyOwner, validNumber` becomes
- `validators: onlyOwner, validNumber` (and those referenced elements must now be declared as `validator ...`)

(Variable name `modifiersKeyword` is now a bit misleading in Grammar 2, but that’s only a model feature name; syntactically it matches the literal `'validators:'`.)

### 4) New `DataStore` rule
Grammar 2 adds:
```xtext
DataStore:
  'dataStore' name=ID 'fromType' fromType=DataType 'toType' toType=DataType;
```
This enables contract instances to declare mappings/typed stores like:
- `dataStore balances fromType address toType integer`

### 5) No change to these parts
- `Attribute`, `Event`, `Param`, `Output` rules remain the same.
- `DataType` enum remains the same.
- `Contract` header fields (`version`, optional `hasReceive`, optional `ownership`) remain the same.
