## Differences: Grammar 2 vs Grammar 1

### 1) Top-level elements inside `Contract`
- **Grammar 1** allowed:
  - `attributes+=Attribute | events+=Event | methods+=Method | modifiers+=Modifier`
- **Grammar 2** allows:
  - `attributes+=Attribute | events+=Event | methods+=Method | validators+=Validator | dataStores+=DataStore`

So:
- `Modifier` is **removed** from the contract body and replaced by `Validator`.
- A new construct `DataStore` is **added** and can appear at contract top-level.

### 2) Renaming / semantic shift: `Modifier` → `Validator`
- **Grammar 1 rule**: `Modifier: 'Modifier' name=ID ...`
  - Keyword was **capitalized**: `'Modifier'`
- **Grammar 2 rule**: `Validator: 'validator' name=ID ...`
  - Keyword is **lowercase**: `'validator'`

Structure is otherwise the same:
- Both have optional params, and a block with:
  - `message: STRING`
  - `validation: STRING`

### 3) `Method` references change (`modifiers:` → `validators:`)
Inside `Method` body:

- **Grammar 1**:
  - `modifiersKeyword='modifiers:' modifiers+=[Modifier] ...`
- **Grammar 2**:
  - `modifiersKeyword='validators:' validators+=[Validator] ...`

So methods now reference **validators** and the literal keyword in the DSL becomes `validators:` instead of `modifiers:`.

Note: the assignment name `modifiersKeyword` is still used in Grammar 2 even though it matches `'validators:'` (a minor naming inconsistency, but it will still work).

### 4) New rule: `DataStore`
Grammar 2 adds:

```xtext
DataStore:
  'dataStore' name=ID 'fromType' fromType=DataType 'toType' toType=DataType
;
```

This introduces a new top-level declaration with:
- a name (`ID`)
- two typed fields (`fromType`, `toType`) using `DataType`.

### 5) Unchanged parts
- `Contract` header structure (`Contract name { version: ... hasReceive? ownership? ... }`) remains the same.
- `Attribute`, `Event`, `Param`, `Output`, and `DataType` enum are unchanged (aside from being followed by the new `DataStore` rule in Grammar 2).
