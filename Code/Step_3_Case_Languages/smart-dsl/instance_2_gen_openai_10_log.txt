### Key differences: Grammar 2 vs Grammar 1

#### 1) Top-level elements allowed inside `Contract`
- **Grammar 1:**
  `attributes | events | methods | modifiers`
- **Grammar 2:**
  `attributes | events | methods | validators | dataStores`

So **`Modifier` is removed** from the contract body alternatives and replaced by **`Validator`**, and a new **`DataStore`** element is added.

#### 2) `Modifier` renamed/replaced by `Validator`
- **Grammar 1 rule:** `Modifier: 'Modifier' name=ID ...`
  - Keyword is capitalized: **`'Modifier'`**
- **Grammar 2 rule:** `Validator: 'validator' name=ID ...`
  - Keyword is lowercase: **`'validator'`**
- Internals are the same structure: parameters, then `{ message: ..., validation: ... }`.

This is a **terminology + concrete syntax change**, and also changes cross-references in methods (see below).

#### 3) Method now references validators instead of modifiers
- **Grammar 1 Method block:**
  - `modifiersKeyword='modifiers:' modifiers+=[Modifier] ...`
- **Grammar 2 Method block:**
  - `modifiersKeyword='validators:' validators+=[Validator] ...`

So in instances:
- `modifiers: onlyOwner, validNumber` becomes
- `validators: onlyOwner, validNumber`

Also note the grammar still names the assigned feature `modifiersKeyword` in Grammar 2, but it matches the literal **`'validators:'`**. (Not wrong, just a slightly misleading feature name.)

#### 4) New `DataStore` construct
Added rule:
```xtext
DataStore:
  'dataStore' name=ID 'fromType' fromType=DataType 'toType' toType=DataType
;
```
This allows declarations like:
`dataStore balances fromType address toType integer`

#### 5) Everything else is effectively unchanged
- `Contract` header, `version`, optional `hasReceive`, `ownership` remain.
- `Attribute`, `Event`, `Param`, `Output`, and `DataType` enum are unchanged (same literals and structure).
- `stateMutability:` optional part of `Method` remains the same.
