### Key differences: Grammar 2 vs Grammar 1

1. **Top-level elements inside `Contract`**
   - **Grammar 1** allows: `Attribute | Event | Method | Modifier`
   - **Grammar 2** allows: `Attribute | Event | Method | Validator | DataStore`
   - So: **`Modifier` was removed/replaced**, and **`DataStore` was added**.

2. **`Modifier` renamed/replaced by `Validator`**
   - **Grammar 1 rule:** `Modifier: 'Modifier' name=ID ...`
     - Keyword is capitalized: **`Modifier`**
   - **Grammar 2 rule:** `Validator: 'validator' name=ID ...`
     - Keyword is lowercase: **`validator`**
   - Structure is otherwise essentially the same:
     - Has `name`, optional parameter list, then a block with:
       - `message: STRING`
       - `validation: STRING`

3. **Method references changed from modifiers to validators**
   - **Grammar 1 (Method block):**
     - `modifiers:` followed by references to `[Modifier]`
   - **Grammar 2 (Method block):**
     - The literal is now **`validators:`** (but note the assignment name is still `modifiersKeyword`—just a naming artifact).
     - It references `[Validator]` instead of `[Modifier]`.
   - Consequence: Instance 1 would need:
     - `Modifier ...` → `validator ...`
     - `modifiers: ...` → `validators: ...`

4. **New concept: `DataStore`**
   - Added rule:
     ```xtext
     DataStore:
       'dataStore' name=ID 'fromType' fromType=DataType 'toType' toType=DataType
     ;
     ```
   - And `Contract` can now contain `dataStores+=DataStore`.
   - This is entirely new in Grammar 2 (no equivalent in Grammar 1).

5. **Everything else is unchanged**
   - `Contract` header fields (`version`, `hasReceive`, `ownership`) remain the same.
   - `Attribute`, `Event`, `Param`, `Output`, and `DataType` enum are unchanged (aside from positioning of `DataStore` after the enum, which is fine in Xtext).
