### Differences: Grammar 2 vs Grammar 1

#### 1) Top-level elements inside `Contract`
- **Grammar 1:** allows `(attributes | events | methods | modifiers)*`
- **Grammar 2:** allows `(attributes | events | methods | validators | dataStores)*`
  - `Modifier` declarations are removed and replaced with `Validator`
  - A new top-level construct `DataStore` is added

#### 2) `Modifier` renamed/reworked to `Validator`
- **Grammar 1:** rule name and keyword are `Modifier` / `'Modifier'`
- **Grammar 2:** rule name and keyword are `Validator` / `'validator'`
- Structure is otherwise the same:
  - name + optional params
  - body with `message:` and `validation:`

#### 3) Method references updated from modifiers to validators
- **Grammar 1 (Method body):**
  - keyword literal: `'modifiers:'`
  - references: `modifiers+=[Modifier]`
- **Grammar 2 (Method body):**
  - keyword literal changed to: `'validators:'`
  - references: `validators+=[Validator]`
- Note: the assignment name `modifiersKeyword` remains, but now captures `'validators:'` (a naming inconsistency, but it will still parse).

Grammar 2 adds:
```
DataStore:
  'dataStore' name=ID 'fromType' fromType=DataType 'toType' toType=DataType;
```
So contracts can declare data stores mapping one `DataType` to another (both restricted to the `DataType` enum).

- `Contract` header, `version`, `hasReceive`, `ownership` are the same
- `Attribute`, `Event`, `Param`, `Output`, and `DataType` enum are unchanged
- `Method` signature format is unchanged (including optional `stateMutability:` and optional outputs list after `->`)
